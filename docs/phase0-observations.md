# Phase 0 Observations

## 我們做出了什麼？

- 把 M-001 整理成「可能是清泥需求的候選訊息」，但明確保留為需要人工確認。
- 把 M-001 的主要問題標示出來：來源不確定、地點模糊、需求規模不明、沒有確認人與確認時間。

## 哪些資料讓 agent 很容易整理？

- M-001 內容有「清泥」「十幾個人」這類明顯的任務線索，容易讓人直覺認為它是可執行需求。

## 哪些資料讓 agent 亂補或誤判？

- 如果把 M-001 當成已確認任務，會誤把一則口耳訊息當成事實；這會把模糊地點與未核實的人力需求直接寫成可派工內容。

## 哪些資訊不能直接變成任務？

- M-001 目前不能直接變成任務，因為它只有模糊描述，沒有確認的地址、需求來源、責任單位與是否真的需要清泥。

## 哪些資訊的操作者不是當事人？

- M-001 的訊息來源是社群貼文／口述說法，並不是現場確認的正式需求單；它更像是「有人說有這個需求」而不是「已被確認的任務」。

## 我們剛剛偷偷假設了什麼？

- 我們不能假設「光復車站後方」就是唯一且可定位的現場，或假設「十幾個人」就是確切人力需求。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要現場志工或站務人員把 M-001 的地點、需求緣由與是否真的需要清泥再確認一次。

## 我們需要更清楚資料格式的地方

- 需要把「需求類型」「地點精度」「來源可信度」「是否已被現場確認」這些欄位更清楚地呈現。

## 我們需要人工確認的地方

- M-001 需要人工確認：地點是否真在老雜貨店後方、是否確實需要清泥、是否有足夠資訊可供後續派工。
